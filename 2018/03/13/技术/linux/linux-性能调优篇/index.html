优化性能的概述

性能问题，主要实现的是linux操作系统和应用程序的最佳结合。系统的性能是指操作系统完成任务的有效性、稳定性和响应速度。系统的性能与上下游的多个方面都密切相关，任何一个环节出现问题，都会影响整个系统的性能。

因此当linux应用现出问题时，应当从应用程序、操作系统、服务器硬件、网络环境等多个方面综合有序的排查，定位问题出现在哪个部分，然后对症解决。

在应用程序、操作系统、服务器硬件、网络环境等方面，影响性能最大的是应用程序和操作系统两个方面，因为这两个方面出现问题不易察觉，隐蔽性强。而硬件、网络方面只要出现问题，一般都能马上定位。

影响性能的因素

1.CPU
2.内存
3.磁盘I/O
4.网络I/O

  • CPU
    对应于需要查询操作的服务，比如：邮件服务器、动态网站等等
  • 内存
    对应于需要缓存数据的服务，比如：打印服务器、数据库服务器、静态网站等等，并且在内存问题上，虽然有物理内存和虚拟内存的说法，但当应用使用虚拟内存时会降低性能，而且在内存问题上要注意CPU和操作系统的架构，如果是32位，最大也只能使用4G内存，此时过多内存将是浪费。
  • 硬盘I/O
    对应于有频繁读写的应用，如果磁盘I/O性能得不到满足，会导致应用的停滞。

以上各个方面之间都是相互依赖的，不能孤立的从某个方面来排查问题。当一个方面出现性能问题时，往往会引发其他方面出现问题。

比如：

  • 大量的磁盘读写势必消耗CPU和磁盘I/O资源
  • 内存的不足会导致频繁地进行内存页写入磁盘、磁盘写到内存的操作，造成磁盘I/O瓶颈
  • 大量的网络流量也会造成CPU过载

所以，在处理性能问题时，要纵观全局，从各个方面进行综合考虑，并且对于这些硬件层面的问题，当达到瓶颈，解决方法只能是提高硬件配置

解决问题的思路

1.外部网络环境
2.服务器硬件设备
3.操作系统

  • 网络
  • CPU
  • 内存
  • 硬盘I/O

4.应用程序自身

性能优化是一个系统工程，四个要素之间相互关联，要综合起来分析，上下游与关联周边一起分析。

CPU性能评估

vmstat

输出说明：
1.procs

  • r：表示运行和等待CPU时间片的进程数，这个值如果长期大于系统CPU的个数，就说明CPU不足，需要增加CPU
  • b：表示在等待资源的进程数，比如正在等待I/O或内存交换等

2.memory

  • swpd：表示切换到内存交换区的内存数量（以KB为单位）。如果swpd的值不为0，或比较大，而且si、so的值长期为0，那么这种情况下一般不用担心，不会影响系统性能。
  • free：表示当前空闲的物理内存数量（以KB为单位）
  • buff：表示buffers cache的内存数量，即缓冲。一般对块设备的读写才需要缓冲
  • cache：表示page cached的内存数量，一般作为文件系统cached，即缓存。频繁访问的文件都会被cached。如果cache值较大，就说明cached的文件数量较多，如果此时IO中的bi比较小，就说明文件系统效率比较好。

3.swap

  • si：表示由swap调入内存
  • so：表示由内存调入swap，一般情况下，si、so的值都为0，如果si、so的值长期不为0，则表示系统内存不足，需要增加系统内存

4.IO
显示磁盘读写状况，这里设置的bi+bo的参考值为1000，如果超过1000，而且wa值较大，则表示系统磁盘IO有问题，应该考虑提高磁盘的读写性能。

  • bi：表示从块设备读入数据的总量（即读磁盘，每秒KB）
  • bo：表示写入到块设备的数据总量（即写磁盘，每秒KB）

5.system
显示采集间隔内发生的中断数，这两个值越大，会看到由内核消耗的CPU时间就越多

  • in：表示在某一时间间隔中观测到的每秒设备中断数
  • cs：表示每秒产生的上下文切换次数

6.CPU
显示了CPU的使用状态，us+sy的参考值为80%，如果us+sy大于80%，就说明可能存在CPU资源不足的情况

  • us：用户进程消耗的CPU时间百分比。us的值比较高时，说明用户进程消耗的CPU时间多，但是如果长期大于50%，就需要考虑优化程序或算法
  • sy：内核进程消耗的CPU时间百分比。sy的值较高时，说明内核消耗的CPU资源很多。
  • id：CPU处于空闲状态的时间百分比
  • wa：IO等待所占用的CPU时间百分比。wa值越高，说明IO等待越严重。根据经验，wa的参考值为20%，如果wa超过20%，就说明IO等待严重，引起IO等待的原因可能是磁盘大量随机读写造成的，也可能是磁盘或磁盘控制器的带宽瓶颈造成的（主要是块操作）

总结：在CPU的评估中，需要重点注意的是procs项r列的值和cpu项中us、sy和id列的值。

sar -u

sar命令需要另外安装sysstat包，sar命令可以对系统的每个方面进行单独的统计，但是使用sar命令会增加系统开销，不过这些开销是可以忽略的，对系统的统计结果不会有很大影响。

sar命令对CPU统计输出：（这个输出是对系统整体CPU使用状况的统计，每项的输出都非常直观，并且最后一行Average是个汇总行，是对上面统计信息的一个平均值）

输出说明：

  • %user：用户进程消耗的CPU时间百分比
  • %nice：运行正常进程所消耗的CPU时间百分比
  • %system：系统进程消耗的CPU时间百分比
  • %iowait：IO等待所占用的CPU时间百分比
  • %steal：在内存相对紧张的环境下pagein强制对不同的页面进行的steal操作
  • %idle：CPU处在空闲状态的时间百分比

在一个多CPU的系统中，如果程序使用了单线程，就会出现一个现象，CPU的整体使用率不高，但是系统应用响应缓慢，这可能是程序使用单线程的原因。单线程只使用一个CPU，导致这个CPU占用率为100%，无法处理其他请求，而其他的CPU却闲置，这就导致了整体CPU使用率不高，而应用缓慢现象的发生。

针对这个问题，可以对系统的每个CPU分开查询，统计每个CPU的使用情况。

这个输出是对系统的第1颗CPU的信息统计。需要注意的是，sar中对CPU的计数是从0开始的，因此，sar -P 0 3 5表示对系统的第一颗CPU进行每隔3秒一次，共5次的信息统计。sar -P 4 3 5则表示对系统的第5颗CPU进行每隔3秒一次，共5次的信息统计。依次类推。从sar的首行输出中也可知，此系统有4颗CPU。

iostat -c

iostat主要用于统计磁盘IO状态，但是也能查看CPU的使用信息，它的局限性是只能显示系统所有CPU的平均信息。

这里使用了-c选项，只显示系统CPU的统计信息，输出中每项代表的含义与sar命令的输出项完全相同。
（可以说iostat秉持着vmstat的全局信息统计，又拿着sar的输出菜单项）

uptime

uptime用于统计系统当前的负载状况，输出的信息依次为系统现在的时间，系统从上次开机到现在运行了多长时间，系统目前有多少登录用户，系统在1分钟内、5分钟内、15分钟内的平均负载。

这里需要注意的是load average这个输出值，即平均负载，3个输出值的大小一般不能大于系统CPU的个数。例如此系统有4个CPU，如果load average的3个值长期大于4时，说明CPU很繁忙，负载很高，可能会影响系统性能。但是偶尔大于4时，倒不用担心，一般不会影响系统性能。相反，如果load average的输出值小于CPU的个数，则表示CPU还有空闲的时间片。

总结：以上这些命令只能查看CPU是否繁忙，负载是否过大，但是无法知道CPU为何负载过大，因此，判断系统CPU出现问题后，要结合top、ps等命令进一步检查是由哪些进程导致CPU负载过大的。引起CPU资源紧缺的原因可能是应用程序不合理造成的，也可能是硬件资源匮乏引起的，所以，要具体问题具体分析，或优化应用程序，或增加系统CPU资源。

内存性能评估

free

free -m表示以MB为单位查看内存使用情况。重点关注的项是free（空闲）、buffers（缓冲）、cached（缓存）。由图可知，此系统空闲内存有2945MB，缓冲了288MB，缓存了1222MB的文件和目录。对于应用程序而言，可用内存有4457MB，其包含了free（空闲）、buffers（缓冲）和cached（缓存）。另外swap交换分区还未使用，所以此系统内存资源还非常充足。

一般有这样一个经验公式：应用程序可用内存/系统物理内存>70%时，表示系统内存资源非常充足，不影响系统性能，应用程序可用内存/系统物理内存<20%时，表示系统内存资源紧缺，需要增加系统内存，20%<应用程序可用内存/系统物理内存<70%时，表示系统内存资源基本能满足应用需求，暂时不影响系统性能

free -s可以在指定的时间段内不间断地监控内存的使用情况

watch+free

通过watch可以动态地监控命令的运行结果，省去手动执行的麻烦

比如：watch -n 3 -d free

其中-n指定重复执行的时间，-d表示高亮显示变动

vmstat

vmstat中重点关注的是swpd、si、so行。

sar -r

输出说明：

  • kbmemfree：空闲物理内存大小
  • kbmemused：已使用物理内存大小
  • %memused：已使用内存占总内存大小百分比
  • kbbuffers：buffer cache，即缓冲大小
  • kbcached：page cache，即缓存大小
  • kbcommit：应用程序当前使用的内存大小
  • %commit：应用程序当前使用的内存百分比

sar的输出其实与free的输出完全对应，不过sar更加人性化，不仅给出了内存使用量，还给出了内存使用的百分比以及统计的平均值，从%commit项可知，此系统目前内存资源充足。

磁盘I/O性能评估

磁盘I/O性能评估的准则：

  • 选RAID，根据应用的不同，选择不同的RAID方式
    • 经常有大量读操作，选择RAID5
    • 经常有大量写操作，选择RAID0
    • 对数据安全要求较高，同时对读写也有要求，选择RAID01
  • 用内存，尽可能用内存的读写代替直接磁盘I/O，使频繁访问的文件或数据放入内存中进行操作处理，因为内存的读写操作比直接磁盘读写的效率要高千倍
  • 区分文件，将经常进行读写的文件与长期不变的文件独立出来，分别放置到不同的磁盘设备上
  • 祼设备，对于写操作频繁的数据，可以考虑使用裸设备代替文件系统。（祼设备的优点就是文件系统的缺点，反之也是如此，这就需要我们根据自己的业务需要做出合理的规划和衡量）
    • 使用祼设备的优点有：
      • 数据可以直接读写，不需要经过操作系统级的缓存，节省了内存资源，避免了内存资源争用
      • 避免了文件系统级的维护开销，比如：文件系统需要维护超级块，inode等
      • 避免了操作系统的cache预读功能，减少了I/O请求。
    • 使用祼设备的缺点有：
      • 数据管理、空间管理不灵活
      • 需要很专业的人来操作

sar -d

输出说明：

  • DEV：磁盘设备名称
  • tps：每秒到物理磁盘的传送数，也就是每秒的I/O流量，一个传送就是一个I/O请求，多个逻辑请求可以被合并为一个物理I/O请求
  • rd_sec/s：每秒读取设备的扇区数（1扇区=512字节）
  • wr_sec/s：每秒写入设备的扇区数
  • avgrq-sz：平均每次设备I/O操作的数据大小（以扇区为单位），类似于超市排队中每人所买东西的多少
  • avgqu-sz：平均I/O队列长度，类似于超市排队中单位时间内平均排队的人数
  • await：平均每次设备I/O操作的等待时间（以毫秒为单位），类似于超市排队中每人的等待时间
  • svctm：平均每次设备I/O操作的服务时间（以毫秒为单位），类似于超市排队中收银员的收款速度
  • %util：一秒中有百分之几的时间用于I/O操作，类似于超市收银台前有人排队的时间比例

对于磁盘I/O性能，一般有如下评判标准：

  • svctm，正常情况下，svctm应该小于await值的，（收银员肯定慢于排队的人数，要不然就不用排队了呀），而svctm的大小和磁盘性能有关，CPU、内存的负荷也会对svctm值造成影响，过多的请求也会间接导致svctm值的增加
  • await，await值的大小一般取决于svctm的值和I/O队列长度以及I/O请求模式。如果svctm的值与await很接近，就表示几乎没有I/O等待，磁盘性能很好，如果await值远高于svctm的值，则表示I/O队列等待太长，系统上运行的应用程序将变慢，此时可以通过更换更快的硬盘来解决问题。
  • %util，%util也是衡量磁盘I/O的一个重要指标，如果%util接近100%，就表示磁盘产生的I/O请求太多，I/O系统已经满负荷地在工作，该磁盘可能存在瓶颈，长期下去，势必影响系统性能，可以通过优化程序或通过更换更高、更快的硬盘来解决此问题

iostat -d

输出说明：

  • Device：磁盘设备名称
  • tps：每秒I/O流量
  • Blk_read/s：每秒读取的数据块
  • Blk_wrtn/s：每秒写入的数据块
  • Blk_read：读取的所有块数
  • Blk_wrtn：写入的所有块数

上面输出的第一项是系统从启动以来到统计时的所有传输信息，从第二次输出的数据才代表在检测的时间段内系统的传输值

可以通过Blk_read/s和Blk_wrtn/s的值对磁盘的读写性能有一个基本的了解。

  • 如果Blk_wrtn/s的值很大，就表示磁盘的写操作很频繁，可以考虑优化磁盘或优化程序，
  • 如果Blk_read/s的值很大，就表示磁盘直接读取操作很多，可以将读取的数据放入内存中进行操作。
iostat -x

iostat -x提供了对每个磁盘的单独统计，如果不指定磁盘，默认就是对所有磁盘进行统计。

这个输出基本与sar -d相同，输出说明如下：

  • rrqm/s：每秒进行merged的读操作数目
  • wrqm/s：每秒进行merged的写操作数目
  • r/s：每秒读I/O设备的次数
  • w/s：每秒写I/O设备的次数
  • rsec/s：每秒读取的扇区数
  • wsec/s：每秒写入的扇区数

vmstat -d

总结：衡量磁盘I/O好坏是多方面的，有应用程序本身、硬件设计上的、系统自身配置的问题等。要解决I/O的瓶颈，关键是要提高I/O子系统的执行效率。
例如：

  • 从应用程序对磁盘读写进行优化，能够放到内存执行的操作，尽量不要放到磁盘
  • 硬件设计上，对磁盘存储方式进行合理规划，选择适合自己的RAID存取方式
  • 系统自身配置上，可以选择适合自身应用的文件系统，必要时使用祼设备提高读写性能

网络性能评估

ping

通过ping命令检测网络的连通性。

time值显示了两台主机之间的网络延时情况，如果此值很大，则表示网络的延时很大，单位为毫秒。在输出的最后，是对上面输出信息的一个总结，packet loss 表示网络丢包率，此值越小，表示网络的质量越高

netstat -i

检测网络接口状况

输出说明：

  • Iface：网络设备的接口名称
  • MTU：最大传输单元，单位字节
  • Met：
  • RX-OK：完整接收多少数据包
  • RX-ERR：接收数据包时发生了多少错误
  • RX-DRP：接收数据包时丢弃了多少数据包
  • RX-OVR：接收数据包时遗失了多少数据包
  • TX-OK：完整发送多少数据包
  • TX-ERR：发送数据包时发生了多少错误
  • TX-DRP：发送数据包时丢弃了多少数据包
  • TX-OVR：发送数据包时遗失了多少数据包
  • Flg：接口标记
    • L：表示该接口是回环设备
    • B：表示设置了广播地址
    • M：表示接收所有数据包
    • R：表示接口正在运行
    • U：表示接口处于活动状态
    • O：表示在该接口上禁用arp
    • P：表示一个点到点的连接

正常情况下，RX-ERR/TX-ERR、 RX-DRP/TX-DRP、RX-OVR/TX-OVR的值都应该为0，如果这几个选项的值不为0，并且很大，那么网络质量肯定有问题，网络传输性能也一定会下降。

当网络传输存在问题时，可以检测网卡设备是否存在故障：

  • 网卡接触不良，重新接入以修复
  • 网卡本身坏了，更换新的网卡
  • 网卡带宽太低，更换千兆或光纤口
  • 网络部署环境是否合理，网络设备环境、机房环境、主干网络环境

netstat -r

检测路由表信息

在网络不通，或网络异常时，ping不通时，首先想到就是检查系统的路由表信息，防火墙信息。这里我们说说路由表。
netstat -r的输出结果与route命令的输出完全相同。重点关注default行对应的值，因为其指向的是默认网关出口

sar -n

显示系统的网络运行状态，主要是一些发送、接收的数据量信息。

sar提供4种不同的选项来显示网络统计信息，通过-n选项可以指定4个不同类型的开关，即DEV、EDEV、SOCK、FULL。

  • DEV：显示网络接口信息
  • EDEV：显示网络错误的统计数据
  • SOCK：显示套接字信息
  • FULL：显示所有3个开关
    xxxxx
    输出说明：
  • IFACE：网络接口设备
  • rxpck/s：每秒钟接收的数据包大小
  • txpck/s：每秒钟发送的数据包大小
  • rxkB/s：每秒钟接收的字节数
  • txkB/s：每秒钟发送的字节数
  • rxcmp/s：每秒接收的压缩数据包
  • txcmp/s：每秒发送的压缩数据包
  • rxmcst/s：每秒接收的多播数据包

traceroute

跟踪数据包的传输路径

nslookup

用于判断DNS解析信息

总结：网络问题的解决方法一般是增加网络带宽或优化网络部署环境。
