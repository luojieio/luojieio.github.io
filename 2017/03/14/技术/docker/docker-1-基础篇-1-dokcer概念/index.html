docker-1.基础篇-1.dokcer概念

发表于 2017-03-14 | 分类于 技术 ， docker |



官方帮助

更新至docker 1.12.3

docker是什么？

  • docker是利用操作系统本身已有的机制和特性（命名空间name space、控制组cgroups、chroot）实现的基于LXC容器技术的改进型容器技术。docker是基于LXC容器模块实现，如同KVM是基于KVM模块实现类似。docker引擎的基础是linux容器技术（linux containers，LXC），容器有效地将由单个操作系统管理的资源划分到孤立的组中，以便更好地在孤立的组之间平衡有冲突的资源使用需求。与虚拟化相比，这样既不需要指令级模拟，也不需要即时编译。容器可以在核心CPU本地运行指令，而不需要任何专门的解释机制。此外，也避免了准虚拟化和系统调用替换中的复杂性。
  • docker没有发明容器，它仅是容器的改进者，其本身并不提供容器技术，它仅是使得容器更易于使用。（就像农夫山泉广告写的那样，我们不生产水，我们只是大自然的搬运工）。
  • 容器不是虚拟化。在没有docker的时代，商家通常使用硬件虚拟化（也称为虚拟机）以提供隔离。其缺点就是：耗时（通常以分钟记）、耗资源（除了应用本身所必需的资源外，还有整个操作系统副本的资源消耗）。docker容器不使用硬件虚拟化。运行在docker容器中的程序接口和主机的linux内核直接打交道。（每个docker容器都是作为docker守护进程的子进程而存在的，所以每个容器就像操作系统上运行的一个个进程，你能说父进程和子进程是使用了虚拟化技术吗？显然不是。）它不像虚拟化技术，还需要中间层进行转换，容器中运行的程序和计算机操作系统之间没有额外的中间层，没有资源被冗余软件的运行或虚拟硬件的模拟而浪费掉。因此docker不是一个虚拟化技术，相反，它可以帮助使用已经内置到操作系统中的容器技术。
  • docker是基于Go语言实现的开源项目，诞生于2013年初，最初发起者是dotCloud公司。dotCloud公司后来也改名为Docker Inc，专注于Docker相关技术和产品的开发。docker项目目前已加入了linux基金会，遵循Apache 2.0 协议，全部开源代码均在https://github.com/docker/docker 上进行维护
  • docker的思想是一次封装（打包），到处运行，docker为应用的开发和部署提供了一站式的解决方案。（可以直接将整个环境部署）。题外话：python的思想是一次开发，到处运行 。
  • docker采用C/S架构。也就是说它有客户端client和服务端server即daemon守护进程，因此我们可以远程来管理容器。
  • docker通过镜像的方式来实现分发容器，docker镜像是一个容器中运行程序的所有文件的捆绑“快照”，即镜像是不变的，容器是快照，只要你想，你就可以从镜像中创建尽可能多的容器，但是当你这样做时，从相同镜像启动的容器不共享文件系统的更改。分发软件，就是分发这些镜像，并在接收的机器上创建容器。（镜像和容器的关系就像程序和进程的关系，或镜像就像只读光盘）。镜像在docker的生态系统中是可交付的基本单位。为了便于分发，docker提供了一组简化分发镜像的基础设施组件。
    包括：
    • 注册服务器（下有仓库，即目录）
    • 索引服务器
      你可以使用docker官方的，也可以使用第三方托管的，也可以自建。

容器是什么？

linux容器其实不是一个全新的概念。最早的容器技术可以追溯到1982年UNIX系列操作系统上的chroot工具
早期的容器实现技术包括：

  • FreeBSD的FreeBSD jail （2000年发布）
  • GNU/Linux的linux-vserver （2001年10月） http://linux-vserver.org/
  • Sun Solaris的Solaris Containers （2004年发布）
  • OpenVZ （2005年） http://openvz.org

早期的linux容器技术没有集成在linux内核标准中，现在的LXC已经集成在linux的内核标准中，不需要再另外打补丁来实现容器化。LXC是以模块的形式集成的。

从LXC到docker

在LXC的基础上，docker进一步优化了窗口的使用体验，docker提供了各种容器管理工具让用户无需关注底层的操作，可以简单明了地管理和使用容器。
可以简单的将docker容器理解为一种沙盒（sandbox），每个容器内运行一个应用，不同的容器相互隔离，容器之间也可以建立通信机制，容器的创建和停止都十分快速，容器自身对资源的需求也十分有限，远远低于虚拟机，很多时候，甚至直接把容器当作应用本身也没有任何问题。因为它相当于把应用的整个部署环境打包封装在一个镜像中了。

为什么要使用docker？

即使用docker的好处或者说docker能解决的问题：

  • 隔离工具。保持一台整洁的机器，因为你很难在一开始就编制好一切，docker可以做为一个隔离工具，解决软件之间的依赖关系使其组织有序。不用再担心移除某个软件或升级某个软件所带来的连锁反应不可控问题。
  • 简单工具。只做必需之事，使用一个简单易用的工具来解决那些复杂或费时的工作，不用规划资源，修改配置，不同平台安装，不同的启动方式 等等，直接一条命令就可用了。
  • 安全工具。测试新东西，但担心是否是恶心代码。docker可以做为一个安全工具。受益于docker的隔离机制，哪怕你的应用出现了安全隐患或漏洞也不会影响到你的宿主机，因为它的影响仅受限于容器内的那一亩三分地。
  • 软件管理工具集。替代yum或rpm式的包管理工具。
  • 方便编写与发布软件。通过docker发布，用户会更容易安装并运行它。
  • 规范开发环境。在docker封装的开发环境中编写软件将节省配置或共享环境的时间，因为从软件的角度来看，每个环境都是一样的。
  • 持续集成与持续部署。创建build、测试和部署管道会变得很简单。
  • 软件分发供应商。docker可以作为软件分发供应商，像应用商店一样，一键下载部署，即下即用，不用额外配置。
  • 多平台兼容提高应用可移植性。docker可以运行N种软件，一次构建处处运行，多平台兼容，你的应用不需要二次开发就可以方便的移植到其它平台上。
  • 提升排错效率。哪怕你的应用出现了故障，没关系，停了，重新开一个容器就会完好如初，直接将排错时间压缩为0，就像苹果的售后政策一样，坏了直接换新的，还修什么。
  • 随时随地获取应用。docker在正确的地点、正确的时间顺应了正确的趋势–即高效地构建应用。现在开发者需要能方便的创建运行在云平台上的应用，也就是说应用必须能够脱离底层机器，而且同时必须是任何时间任何地点可以获取并运行。
  • 方便部署与迁移。通过容器来打包应用，意味着迁移只需要在新的服务器上启动需要的容器就可以了。节约了大量的时间，降低部署过程出现问题的风险，不会再出现，我的电脑运行正常的问题了。
  • 更高效的资源利用。不需要额外的虚拟化管理程序VMM和hypervisor的支持，由于它是内核级的容器化，所以可以实现更高的性能，同时对资源的额外需求很低。
  • 更简单的更新管理。使用Dockerfile，只需要小小的配置修改，就可以替代以往大量的更新工作。并且所有修改都是以增量的方式进行分发和更新，从而实现了自动化（配置文件式）并且高效（增量）的容器管理。

容器化VS虚拟化

docker与虚拟机特性比较

  • docker容器很快，启动和停止可以在秒级实现，比虚拟机快得多（怪我咯，人家是分钟级）
  • docker容器对系统资源需求少，一台主机上可以同时运行数千个docker容器。docker容器除了运行其中的应用之外，基本不消耗额外的系统资源，保证应用性能的同时，尽量减小系统开销。传统虚拟机方式运行N个不同的应用就要启动N个虚拟机（每个虚拟机需要单独分配独占的内存、磁盘、CPU等资源），而docker只需要启动N个隔离的容器，并将应用放到容器内即可
  • docker通过类似git的操作来方便用户获取、分发和更新应用镜像，指令简明，学习成本低
  • docker通过Dockerfile配置文件来支持灵活的自动化创建和部署机制，提高工作效率
  • 在隔离性方面，传统虚拟机方式多了一层额外的隔离（操作系统的文件隔离），但docker利用linux系统上的多种防护机制实现了严格可靠的隔离，docker引入了安全选项和镜像签名机制，极大的提高了docker的安全性。传统虚拟机是“操作系统”级的完全隔离，docker是利用“权限机制”级的安全隔离。
