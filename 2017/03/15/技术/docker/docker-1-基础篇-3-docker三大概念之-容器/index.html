容器（Container）

类似于一个轻量级的沙箱，docker利用容器来运行和隔离应用

基于同一个镜像可以创建多个容器
容器是从镜像创建的应用运行实例，所不同的是它带有额外的可写文件系统层。可以将其启动、开始、停止、删除，而这些容器都是相互隔离、互不可见的。

可以把容器看成一个简易版本的linux系统环境（包括了root用户权限、进程空间、用户空间和网络空间等），以及运行在其中的应用程序打包而成的应用盒子

镜像自身是只读的，容器从镜像启动的时候，docker会在镜像的最上层创建一个可写文件系统层，镜像本身将保持不变。

实际上，既可以使用docker load命令来导入镜像存储文件到本地的镜像库，

此外从容器快照文件导入时可以重新指定标签等元数据信息。

在使用-d参数时，容器启动后会进入后台，用户无法看到容器中的信息。

总结

Docker exec与Docker attach 的比较
不论是开发者还是运维人员，都经常有需要进入容器的诉求。
目前看，主要的方法不外乎以下几种：

  1. 使用ssh可以远程或从近程（宿主机）登录进容器
  2. 使用nsenter、nsinit等第三方工具
  3. 使用docker本身提供的工具：attach、exec

方法1：需要在容器中启动sshd，存在开销和攻击面增大的问题，同时也违反了docker所倡导的一个容器一个进程的原则
方法2：需要额外学习使用第三方工具
方法3：基于上两种方法的情况，所以大多数情况最好还是使用docker原生方法，docker目前主要提供了dokcer attach和 docker exec 两个命令。

  • attach是docker 1.3版本之前所提供的，功能较弱，登录后需要按一下回车，并且所登录的会话是公用的，意思就是说无论你attach多少个会话，显示的内容都是同步的，而且exit退出登录，容器也会停止

  • exec 是docker 1.3版本之后推出的新一代执行命令方式，推荐有近程登录需求的使用这种方式，它所登录所使用的会话是独立的，而且从容器中退出时，容器不会停止。

  • 关于重连容器中attach与exec有很大的不同

    • 首先，请知道一点的是，容器是可以重连的，并不是说只有带run -ti 后接/bin/bash这种带终端接口的才可以重新连接进入容器内部。
    • 其次，我们需要知道的是无论是普通应用，如apache，nginx，ssh等等，还是/bin/bash这种虚拟控制台应用，它们在容器里都只是一个普通的应用，都是平等的，归结到底都只是一个进程这个概念很重要。

区别在于：

工具 可否重连
attach 不可以重连
exec 可以重连
ssh 可以重连

当使用attach时，如果容器里只运行了一个非/bin/bash这种对外提供访问接口的应用进程，这时由于没有可供接入的接口程序，所以attach也就无法进入容器里

而当使用exec时，exec的作用是在容器里执行命令，也就是说哪怕容器里原本运行着一个非/bin/bash这种应用进程，我们也可以使用exec再在容器里执行一个/bin/bash的命令，这样容器里就会新增加了一个进程，而这个进程如果恰好又是/bin/bash这种对外提供接入的应用进程时，它也就可以实现为我们进入容器内部提供接口，所以说exec也就是实现了重连的需要。这也是为什么当使用exec进入容器时，当我们退出容器，容器依然不会EXIT，这是因为容器原本运行着的应用进程并没有退出，也就不会导致容器EXIT退出了。

  • 单个进程如果是长期执行的，如httpd当它执行完，进程退出，容器也退出

情况二：直接在CMD后接脚本

由情况一引申出两个问题？

  1. 如何启动多进程？
    我们CMD只能启动一个进程，而写多个CMD时只认最后一个。
    解决思路：将多个启动进程写在一个脚本里，然后CMD执行这个shell脚本不就行了
  2. 但是用CMD执行shell脚本，脚本执行完了，退出，不也就导致容器也自动退出了吗？
    实际情况：脚本执行完后，按理说应该随之也就退出了，但是实际情况在docker中却并不会导致容器退出，这是为什么呢？

针对于shell脚本执行完了为什么容器不退出，两个猜想：

猜想一：是否只要是写在脚本中，脚本执行完退出，不会导致容器退出？
我们上面情况一时是直接在CMD处写命令的，所以这没什么可疑惑的，进程执行完，进程退出，容器退出。现在摆在我们面前的是shell脚本按理说也是用CMD处执行的，为什么脚本就有特例，脚本完了，容器可以不退出。

于是连想到是不是docker内部定义了，COMMAND处是循环执行的？只要是脚本执行的都不会导致容器退出？为了验证这个，我们将原来的echo直接写在脚本中，然后启动。

猜想二：基于猜想一的结果，COMMAND不是循环的，也并不是只要是脚本，容器就不会退出，那为什么我们还可以将我们需要启动的多个进程放在一个脚本中执行呢？这不应该啊，为什么它不会退出，于是引申出第二个猜想，是否只要脚本内容中启动的进程没有退出，脚本就不会退出，从而容器也就不会退出呢？

扩展2：关于COMMAND的说明

我们查看容器时可以看到COMMAND

  1. 当手动写CMD命令时，你写的是什么COMMAND处显示的就是什么

  2. 当你是使用Dockerfile中写的CMD关键字写的命令时，它会在COMMAND处显示成这样

会发现它会在你写的命令前面加下/bin/sh -c后接你的命令，其实这也就说明了Dockerfile中写的CMD和ENTRYPOINT后接的命令，它们会根据它所允许使用的定义格式，用相对应的解析程序进行执行。
即：
CMD command param1 param2 对应的前面会加/bin/sh -c 来解析执行后面的命令
CMD [ "execute","param1","param2" ] 对应的前面加exec 来解析执行后面的命令

所以通过docker ps -a查看到COMMAND处显示内容，就可以知道Dockerfile它真正底层运行的命令是什么
