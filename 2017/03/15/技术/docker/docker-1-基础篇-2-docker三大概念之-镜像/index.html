镜像（Image）

类似于虚拟机镜像，可以将它理解为一个面向docker引擎的只读模板，包含了文件系统。（如果不能理解，就把它理解为DVD光盘，只读的，不可重复擦写的光盘）

镜像可以是ubuntu操作系统环境，称为ubuntu镜像，也可以安装了apache应用程序，称为apache镜像

镜像是创建docker容器的基础。docker运行容器前需要本地存在对应的镜像，如果镜像不存在本地，docker会尝试先从默认镜像仓库下载（默认使用docker hub公共注册服务器中的仓库），用户也可以通过配置，使用自定义的镜像仓库

通过pull命令拉取下载过程可知，镜像文件一般由若干层组成，行首的fd74137ff5ae这样的字串代表了各层的ID，下载过程会获取并输出镜像的各层信息。层（layer）其实是AUFS（Advanced Union File System，高级单元文件系统）一种联合文件系统中的重要概念，是实现增量保存与更新的基础。（这里埋下伏笔，建议我们将不变动的内容写入镜像的各个分层，这样更新时就不用更新底层，也节约的镜像的构建时间与容量）

命令

含义 命令 备注
获取镜像 docker pull NAME[:TAG] TAG相当于版本号， 如果不显式的指定TAG，则默认会
选择latest标签，即下载仓库中最新版本的镜像
创建并运行容器 docker run -ti ubuntu /bin/bash 利用镜像创建并运行一个容器，并且以分配tty有交互的
连接容器，在容器中运行bash应用
列出本地主机上已有的镜像 docker images
为镜像添加新标签 docker tag oldimage newimage 相当于一个别名
获取镜像的详细信息 docker inspect 镜像ID或名称 通常使用该ID有前若干个字符组成的可区分字串
来替代完整的ID，只要可以唯一区分出来就行。
inspect命令返回的是一个JSON格式的消息，
如果我们只要其中一项内容时，可以说使用-f参数来指定，
如：
docker inspect -f .Os docker/whalesay）
搜寻镜像 docker search TERM 使用docker search命令可以搜索远端仓库中”共享“的镜像，
默认搜索docker hub官方仓库中的镜像。
支持的参数有:
--automated=false 仅显示自动创建的镜像
--no-trunc=false 输出信息不截断显示
-s,--stars=0 指定仅显示评价为指定星级(收藏)以上的镜像

返回带有包含关键字的镜像，其中包括：
镜像名字、描述、星级（表示该镜像的受欢迎程度）、
是否官方创建、是否自动创建等。

默认的输出结果将按星级评价进行排序。
官方镜像说明是官方项目组创建和维护的
automated资源则允许用户验证镜像的来源和内容
删除镜像 docker rmi IMAGE [IMAGE...] 其中IMAGE可以为标签或ID
当同一个镜像拥有多个标签的时候，
docker rmi命令只是删除了该镜像多个标签中的指定标签,
并不影响镜像文件。

当你使用ID（也可以是ID能进行区分的部分前缀）
进行删除时，会先尝试删除所有指向该镜像的标签，
然后删除该镜像文件本身。

注意：
当有该镜像创建的容器存在时，镜像文件默认是无法被删除的。
如果想强行删除镜像，可以使用-f参数。
即:docker rmi -f IMAGE

不推荐使用-f参数来强制删除一个存在容器依赖的镜像，
因为这样往往会造成一些遗留问题。
在docker 1.3时强制删除一个有在运行的容器的镜像时，
容器会显示为none的镜像状态，
而在1.6版本后，不会再显示none状态，
镜像名中依然使用原镜像ID。并且，容器关闭后哪怕镜像已删除，
依然可以重新启动，这应该是docker 新增加的状态保存机制，
避免数据丢失做的一个处理。
因此，正确的做法是，先删除依赖该镜像的所有容器，再来
删除镜像。但是当镜像只剩下一个标签的时候就要小心了，
此时再使用docker rmi命令会彻底删除该镜像，
执行删除操作的过程其实就是删除这个镜像文件的所有AUFS层
创建镜像 下面的单独表格
存出镜像到本地文件（文件级） docker save -o 目标命名 镜像文件名 类似于tar包归档的格式。
即：
docker save -o ubuntu.tar ubuntu:latest
从存出的本地文件中再导入
（载入）到本地镜像库（文件级）		 docker load -i ubuntu.tar
或
docker load < ubuntu.tar		 类似于tar包解包的格式。
这将导入镜像以及相关的元数据信息（包括标签等），
可以使用docker images命令来进行查看，注意如果
本地镜像列表中本来就已经有了此镜像，
则再次导入时还是跟原来镜像一样的信息，
即版本和标签等元数据都是一样的，
所以它不会报错，只是不会重复再次导入相同
的内容而已。当你把原来的镜像删除了，
再次导入就可以看到新导入的镜像文件了。
上传镜像到仓库，默认上传到
dockerhub官方仓库（需要登录）		 1.先根据你所要上传的仓库（目录）为镜像创建标签
docker tag test:latest luojie/test:latest

2.上传
docker push luojie/test:latest		 第一次使用时，会提示输入登录信息（用户名、密码、邮箱）
或进行注册也可以提前先登录上去，再push
即：
docker login -u luojie

创建镜像

创建镜像的方法有三种：

  • 基于commit对已有镜像的容器创建
  • 基于本地模板导入
  • 基于Dockerfile创建

基于commit对已有镜像的容器创建

主要选项 含义
-a,–author=”” 作者信息
-m,–message=”” 提交消息
-p,–pause=true 提交时暂停容器运行

步骤：
1.启动一个镜像，并在其中进行修改操作，例如创建一个test文件后退出

2.查看容器（记住容器ID ）

3.使用docker commit命令来提交为一个新的镜像，提交时可以使用ID或名称来指定容器。

基于本地模板导入

也可以直接从一个操作系统模板文件导入一个镜像。
比如：使用OPENVZ提供的模板来创建。
OPENVZ模板的下载地址为：https://openvz.org/Download/template/precreated

步骤：
1.下载一个模板压缩包后，使用命令导入模板

2.查看新导入的镜像，已经在本地存在了

基于Dockerfile创建

Dockerfile是一个文本格式的配置文件，用户可以使用Dockerfile快速创建出自定义的镜像

Dockfile方式构建镜像的实践步骤：

  1. 使用commit对所搭建应用的具体步骤进行实际测试，总结每条语句的顺序
  2. 切记commit语句中没有EXPOSE和CMD可以通过配置获取，所以在实际使用commit构建的镜像进行run时，只能通过-p指定端口，而不能用-P，CMD命令也必须手敲，而这两个值通过Dockfile构建时可以直接写在配置文件中
  3. 以上两点总结完成后，再开始编写Dockerfile的构建文件
基本结构

Dockerfile由一行行命令语句组成，并且支持以#开头的注释行，执行也是由上至下执行，类似于shell脚本，只要把一条条语句由前到后写好即可

Dockerfile分为四个部分：

  • 基于的基础镜像信息
  • 维护者信息
  • 镜像操作指令
  • 容器启动时执行指令

指令

指令的一般格式为INSTRUCTION arguments

指令 格式 说明
FROM FROM <image>或FROM <image>:<tag> 第一条指令必须为FROM指令。
并且，如果在同一个Dockerfile中创建多个镜像时，
可以使用多个FROM指令（每个镜像一次）
MAINTAINER MAINTAINER <name> 用于指定维护者信息
RUN RUN <command>或 RUN ["executable","param1","param2"] 前者将在shell终端中运行命令，即/bin/sh -c ‘command’
后者则使用exec执行，指定使用其他终端可以通过第二
种方式实现，
例如：
RUN ["/bin/bash","-c","echo hello"]
每条RUN指令将在当前镜像基础上执行指定命令，
并提交为新的镜像，当命令较长时，可以使用\转义符来换行
CMD 支持三种格式：
CMD ["executable","param1","param2"]
（用exec执行，可执行程序加参数。推荐方式）
CMD ["param1","param2"]
（提供给ENTRYPOINT的默认参数）
CMD command param1 param2
（用sh -c 执行，命令加参数）		 指定启动容器时的执行命令，每个Dockerfile只能
有一条CMD命令，如果指定多条命令，只有最后一条会被执行。
如果用户启动容器时指定了运行的命令，
则会覆盖CMD指定的命令
EXPOSE EXPOSE <port> [<port>...]
如：
EXPOSE 22 80 8443		 用于告诉docker服务端容器“暴露的端口号”，供互联系统使用。
在启动容器时需要通过-P ，docker会自动分配一个
本地主机端口转发到指定的端口
使用-p 则可以具体指定哪个本地主机端口映射到容器暴露的端口
ENV ENV <key> <value> 就像定义变量一下，变量名与变量的内容。
用于指定一个环境变量，会被后续的
RUN指令使用，并在容器运行时保持。
ADD ADD <src> <dest> 用于将复制指定的<src>到容器中的<dest>，
比如将宿主主机上的文件或目录复制到容器中去
其中<src>可以是Dockerfile所在目录的一个相对路径（文件或目录），
也可以是一个URL，还可以是一个tar文件（自动解压为目录）,
除了多了URL和解压功能外，其余与COPY相同，
ADD会把权限也拷贝进镜像
COPY COPY <src> <dest> 复制本地主机的<src>（为Dockerfile所在目录的相对路径，
文件或目录）
到容器中的<dest>。目标路径不存在时，会自动创建当使用本地目录
为源目录时，推荐使用COPY，
COPY会把权限也拷贝进镜像
ENTRYPOINT 有两种格式：
ENTRYPOINT ["executable","param1","param2"]
（用exec执行， 可执行程序加参数）
ENTRYPOINT command param1 param2
（用sh -c执行 命令加参数）		 用于配置容器启动后执行的命令，并且不可被
docker run 提供的参数覆盖
每个Dockerfile中只能有一个ENTRYPOINT（入口点），
当指定多个ENTRYPOINT时，只有最后一个生效
（跟CMD类似，只不过CMD是容器启动时执行，
ENTRYPOINT是启动后执行）
VOLUME VOLUME ["/data"] 创建一个可以从本地主机或其他容器挂载的挂载点，
一般用来存放数据库和需要保持的数据等
USER USER daemon 指定运行容器时的用户名或UID，后续的RUN也会使用指定用户
当服务不需要管理员权限时，可以通过该命令指定运行的用户，
因为用户对应的就是用户所拥有的权限，所以对用户的控制，
就是对权限的控制。
并且可以在之前创建所需要的用户。
如：
RUN groupadd -r postgres && useradd -r -g postgres postgres
要临时获取管理员权限可以使用gosu，而不推荐sudo
WORKDIR WORKDIR /path/to/workdir 相当于cd命令，用于为后续的RUN、CMD、ENTRYPOINT指令
配置工作目录
可以多个WORKDIR指令，后续命令如果参数是相对路径，
则会基于之前命令指定的路径。
如：
WORKDIR /a
WORKDIR b
WORKDIR c
RUN pwd
则最终路径为/a/b/c
ONBUILD ONBUILD [INSTRUCTION] 用于配置当所创建的镜像作为其他新创建镜像的基本镜像时，
所执行的操作指令说明。
如： Dockerfile使用如下的内容创建了镜像image-A
ONBUILD ADD . /app/src
ONBUILD RUN /usr/local/bin/python-build --dir /app/src

如果基于image-A 创建新的镜像时，新的Dockerfile中使用
FROM image-A指定基础镜像时，
会自动执行ONBUILD指令内容，等价于在后面添加了两条指令，
其实从名字也可以看出来，ONBUILD，
不就是在构建时执行的命令嘛
如：
FROM image-A
#自动运行接下来的命令
ADD . /app/src
RUN /usr/local/bin/python-build --dir /app/src
如果使用了ONBUILD的指令的镜像，推荐在标签中注明，例如：ruby:1.9-onbuild
创建镜像

编写好Dockerfile之后，可以通过docker build命令来创建镜像
基本格式：docker build [选项] 路径
该命令将读取指定路径下（包括子目录）的Dockerfile，并将该路径下“所有内容”（包括构建镜像所需要的所有内容，如脚本），发送给Docker服务端，由服务端来创建镜像。因此一般建议放置Dockerfile的目录为空目录

另外，可以通过.dockerignore文件（每一行添加一条匹配模式）来让Docker忽略路径下的目录和文件

选项 含义
-t 指定镜像的标签信息
