Supervisor：一个进程控制系统

Supervisor基于C/S架构，允许用户在类UNIX系统上监控和控制多个进程，跟它类似的工具还有：launchd,daemontools, 和runit.跟这些程序不同的是，它并不意味着作为”process id 1”替代init。相反，它用于控制与项目或客户相关的进程，并且意在像启动时像其他任何程序一样开始。

概览

它受到以下痛点的启发而诞生的：

方便性
为每个流程实例编写rc.d脚本通常是不方便的。当然，不可否认的是rc.d脚本是最好的和最低层的通用进程载体，它可以初始化/自动启动/管理进程，但是编写脚本和维护可能很痛苦。另外，rc.d脚本不可以自动重启一个宕掉的进程以及许多程序在崩溃时不会正常重新启动。而supervisor则不同，Supervisord启动进程作为其子进程，并且可以配置为在崩溃时自动重新启动它们。它也可以自动配置为在其自身的调用中启动进程。

准确性
在UNIX上的通常很难获得准确的进程up/down状态。Pidfiles经常说谎。Supervisord将进程作为子进程启动，所以它总是知道其子进程的正确的up/down状态，可以方便地查询这些数据。

代表性
用户往往并不需要使用完整的shell来访问机器来确认哪个进程是正在运行的，处于侦听”低层级”TCP端口的进程通常需要作为root用户启动和重新启动（UNIX的不正当功能）。通常情况下，允许”普通权限”人员停止或重新启动此进程是更好的做法，但是为他们提供shell访问通常是不切实际的，并且为他们提供root访问或sudo访问通常是不可能的。也很难向他们解释为什么这个问题存在。如果以root身份启动supervisor，则可以允许”普通权限”用户控制这些进程，而无需向他们解释进程使用权限这个问题的复杂性。Supervisorctl仅允许了非常有限的访问机器的形式，基本上允许用户通过从简单的shell或web UI发出”停止”，”启动”和”重新启动”命令来查看进程状态和控制超级管理控制的子进程。

进程组
进程通常需要分组启动和停止，有时甚至有启动优先级顺序，人们常常难以解释这个问题。supervisor允许您为进程分配优先级，并允许用户通过supervisorctl客户端发出命令，如”start all”，”restart all”，它们以预分配的优先级顺序启动它们。此外，进程可以分组为”进程组”，一组逻辑关联的进程可以作为一个单元停止并启动。

功能

简单
supervisor通过简单的INI风格的配置文件进行配置，易于学习。 它提供了许多进程选项，使您的生活更容易，如重新启动失败的进程和自动日志轮换。

集中

supervisor为您提供一个启动，停止和监控进程的地方。 进程可以单独或分组控制。 您可以配置Supervisor以提供本地或远程命令行和Web界面来进行管理。

高效
supervisor通过fork / exec启动其子进程，子进程不后台运行化（ daemonize）。 操作系统在进程终止时立即发出指示信号给supervisor，而不像一些依赖麻烦的PID文件和定期轮询重启失败进程的解决方案。

扩展
supervisor有一个简单的事件通知协议，用任何语言编写的程序都可以用来监视它，还有一个用于控制的XML-RPC接口。 它还使用可以由Python开发人员利用的扩展点构建。

兼容
supervisor工作在除Windows之外的所有操作系统之上。 它在Linux，Mac OS X，Solaris和FreeBSD上进行了测试和支持。 它完全用Python编写，所以安装不需要C编译器。

久经考验
虽然supervisor今天非常积极地开发，但并不是新的软件。 supervisor已经存在多年，已经在许多服务器上使用。

supervisor 组件

supervisord

服务端组件命名为supervisord,它负责在自己的调用中启动子程序，响应客户端的命令，重新启动已宕掉或退出的子进程，记录子进程的stdout和stderr输出，以及生成和处理与子流程生命周期中的点对应的”事件”。

服务器进程使用配置文件。它通常位于/etc/supervisord.conf中，此配置文件是”Windows-INI”样式的配置文件。通过正确的文件系统权限来保护此文件非常重要，因为它可能包含未加密的用户名和密码。

supervisorctl

supervisor的命令行客户端名称为supervisorctl。它为supervisord提供的功能提供了一个类似shell的界面。从supervisorctl，用户可以连接到不同的supervisord进程，获取控制的子进程的状态，停止和启动子进程，并获取supervisord运行进程的列表。

命令行客户端通过UNIX域套接字或Internet（TCP）套接字与服务器通信。服务器可以断言客户端的用户在允许他执行命令之前应该出现认证证书。客户端进程通常使用与服务器相同的配置文件，但其中具有[supervisorctl]部分的任何配置文件都可以正常工作。

Web Server

如果您启动针对Internet套接字的supervisord，则可以通过浏览器访问具有与supervisorctl相媲美功能的Web用户界面。访问服务器的URL（例如：http://localhost:9001/）来查看和控制进程状态通过这个web接口，前提条件是你已经激活了配置文件中的`[inet_http_server]`部分。

XML-RPC 接口

服务于Web UI的与HTTP服务器类似的提供了一个XML-RPC接口，可用于询问和控制supervisor及其运行的程序。可以查看XML-RPC API文档，通过这些接口方便我们通过其它程序语言来定制我们的功能。

平台要求

supervisor经过测试，已知在Linux（Ubuntu 9.10），Mac OS X（10.4 / 10.5 / 10.6）和Solaris（10个为Intel）和FreeBSD 6.1上运行。 在大多数UNIX系统上可能会运行正常。

在任何版本的Windows下，supervisor都不会运行。

supervisor已知可工作于python2.4或更高版本的python，但不可以工作于任何python3版本。
